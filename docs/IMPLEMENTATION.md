# 007 Bond -- Implementation Details

## Base

Fork of [wireguard-go](https://github.com/WireGuard/wireguard-go) (MIT License). The WireGuard encryption, handshake, and key management are unmodified. 007 Bond adds a bond layer between the TUN interface and the crypto layer.

## Architecture

```
SEND PATH:
  App -> TUN (bond0) -> FEC encode -> WireGuard encrypt -> multi-path send (all interfaces)
  Control packets (probes/NACKs/preset) bypass FEC -- sent raw

RECEIVE PATH:
  multi-path receive -> WireGuard decrypt -> control packet check ->
    FEC decode -> ARQ gap detection (races FEC) -> jitter buffer -> TUN
```

## Bidirectional Multi-Path (v0.4.0)

Multi-path sending works in both directions through two complementary mechanisms:

### Client -> Server (UAPI bond paths)

The client's `add-bond-paths.sh` detects all network interfaces with a route to the server and configures them as bond paths via the WireGuard UAPI socket. Each bond path gets a dedicated UDP socket bound to that interface's IP address. `SendBuffers` sends every packet through all bond path sockets.

```bash
# UAPI command (generated by add-bond-paths.sh):
set=1
public_key=<hex>
clear_bond_endpoints=true
bond_endpoint=203.0.113.1:51820@192.168.1.100    # via eth0
bond_endpoint=203.0.113.1:51820@192.168.1.101    # via wlan0
```

### Server -> Client (auto-discovery)

The server requires zero per-client path configuration. When receiving packets from a peer, `SetEndpointFromPacket` records each unique source address in a `discovered` map on the peer:

```go
// device/peer.go
peer.endpoint.discovered[key] = discoveredEndpoint{
    endpoint: endpoint,
    lastSeen: time.Now(),
}
```

When `SendBuffers` runs, it sends to:
1. The primary endpoint (standard WireGuard)
2. All discovered endpoints (auto-learned from incoming packets)
3. All configured bond paths (UAPI, client-side only)

Endpoints not seen for 60 seconds are evicted from the discovered map.

This design means the server automatically learns all client paths and replies to all of them -- seamless failover without UAPI configuration.

### Policy-Based Routing

When multiple client interfaces share the same subnet (e.g. eth0 + wlan0 on the same home router), Linux's default routing would send all traffic out a single interface regardless of the source IP. `add-bond-paths.sh` creates per-interface policy routes:

```bash
# For interface N:
ip rule add from <local_ip> table <100+N> prio <32700+N>
ip route replace default via <gateway> dev <iface> table <100+N>
ip route replace <subnet>/24 dev <iface> scope link table <100+N>
```

This ensures packets sourced from each interface's IP actually exit through that interface.

## Recovery Chain

Recovery mechanisms fire in this order, each independent:

1. **Multi-path diversity** -- same packet sent on all interfaces. First copy wins. Independent per-path loss means compound loss probability is very low (30% per path = 9% both).

2. **Sliding-window FEC (XOR, W=5)** -- each data packet generates one repair packet covering the last 5 data packets. Single loss recovery in 20ms. Overlapping windows recover bursts up to W-1=4 consecutive losses.

3. **ARQ (NACK-based)** -- fires for every sequence gap immediately, racing FEC in parallel. If FEC recovers the packet, the retransmit is a harmless duplicate. If FEC fails, ARQ is already in flight. Deadline-checked against jitter buffer playout time.

4. **Jitter buffer** -- per-packet playout deadline. Each packet gets `deadline = insertTime + bufferDepth`. Rate-independent -- works for any traffic pattern.

## Presets

| Preset | LatencyBudgetMs | FEC K | FEC M | Jitter Buffer | Use Case |
|--------|----------------|-------|-------|---------------|----------|
| broadcast | 40 | 2 | 2 | 20ms (1 slot) | Live contribution |
| studio | 80 | 2 | 2 | 60ms (3 slots) | Studio links |
| field | 200 | 2 | 4 | 180ms (9 slots) | WiFi + cellular |

Jitter buffer depth = LatencyBudgetMs - (K-1) * PacketIntervalMs.

### Runtime Preset Changes

Presets can be changed at runtime via `POST /api/preset` or `007-bond preset <name>`. When a preset is changed:

1. Local jitter buffer depth is updated immediately
2. A **control packet (type=5)** containing the preset name is sent through the WireGuard tunnel to all peers
3. Each peer receives the control packet and updates only that sender's jitter buffer depth
4. Each peer has an **independent jitter buffer** on the server -- changing one client's preset does not affect others

This enables per-client latency tuning: a field reporter on cellular can use the `field` preset (200ms) while a studio link on managed ethernet uses `broadcast` (40ms), both connected to the same server.

## Wire Formats

### Sliding FEC Data Packet
```
[Type=0x01][Flags=0x00][DataSeq (8 bytes)][IP payload]
```

### Sliding FEC Repair Packet
```
[Type=0x02][Flags=0x00][RepairSeq (8)][WindowStart (8)][WindowSize (1)][XOR data]
```

### Block FEC (Reed-Solomon)
```
[BlockID (2)][Idx (1)][K (1)][M (1)][DataSeq (8)][IP payload or parity shard]
```

### Control Packets
All control packets start with `BlockID=0xFFFF` to distinguish from FEC data:
```
[0xFF][0xFF][Type (1)][K=0][M=0][payload...]
```

| Type | Format | Purpose |
|------|--------|---------|
| 1 (NACK) | `[count (2)][seq1 (8)][seq2 (8)]...` | Request retransmission |
| 2 (Probe) | `[pathID (4)][timestamp (8)]...` | RTT measurement |
| 3 (Echo) | `[pathID (4)][timestamp (8)]...` | RTT response |
| 4 (Retransmit) | `[dataSeq (8)][IP payload]` | Retransmitted packet |
| 5 (Preset) | `[preset name bytes]` | Signal preset change to peer |

Control packets bypass FEC encoding (not assigned dataSeq, not counted in FEC blocks).

## FEC Decoder Gap Detection

The sliding FEC decoder tracks the highest data sequence seen (`nextExpected`). When a data packet arrives with `seq > nextExpected`, all sequences between are reported as missing -- regardless of whether FEC has already recovered them. This ensures ARQ fires for every gap unconditionally, racing FEC.

```go
// In SlidingFECDecoder.Decode():
if seq > d.nextExpected {
    for s := d.nextExpected; s < seq; s++ {
        missing = append(missing, s)  // no d.received check -- race FEC
    }
}
```

## ARQ Flow

1. Receiver detects gap (FEC decoder reports missing sequences)
2. NACKs fired immediately via `nackTracker` -> `sendFunc`
3. FEC recovery runs in parallel (may fill the gap)
4. Sender receives NACK -> looks up original packet in `retransmitBuffer` -> sends retransmit
5. Retransmit arrives as control packet (type=4) -> inserted into jitter buffer with `sourceARQ`
6. If FEC already recovered it, retransmit is a duplicate (harmlessly discarded)

Deadline check: before sending a NACK, verify at least one path has RTT < remaining time until playout deadline. If not, skip (`arq_deadline_skip` counter).

## Per-Path Health

Each path tracks:
- **RTT** via probe/echo packets (EWMA)
- **Jitter** from RTT variance
- **Loss** via sliding window
- **Burst loss** count and max burst length
- **State machine**: healthy -> degraded -> unstable -> failed -> recovering

## Bond Path Configuration

### Client side (UAPI)

```bash
# add-bond-paths.sh generates UAPI commands:
set=1
public_key=<hex>
clear_bond_endpoints=true
bond_endpoint=server:51820@eth0_ip
bond_endpoint=server:51820@wlan0_ip
```

The client's path monitor timer (`007-bond-paths.timer`) re-scans interfaces every 30 seconds. Only interfaces with a route to the server are added as bond paths. Policy-based routing rules are created for interfaces sharing the same subnet.

### Server side (automatic)

No configuration needed. The server's `SetEndpointFromPacket` in `device/peer.go` tracks all source addresses from incoming packets. `SendBuffers` sends to all discovered endpoints plus the primary endpoint.

## SIP/RTP Routing

### BOND_ROUTES

The `BOND_ROUTES` environment variable in `/etc/007/.env` specifies hosts to route through the bond tunnel:

```bash
BOND_ROUTES=sip.rtegroup.ie,54.220.131.205
```

On startup, `setup-wg.sh`:
1. Resolves each hostname to an IP
2. Appends each IP to WireGuard's `allowed-ips` (so WireGuard accepts return traffic from those IPs)
3. Creates `ip route replace <IP>/32 via <tunnel_gw> dev bond0` entries

### Gateway mode

When the 007 server is separate from the SIP/media server, gateway mode enables NAT so tunnel clients can reach the internet:

```bash
sudo 007-bond gateway on
```

This creates:
- `sysctl net.ipv4.ip_forward=1` (persisted in `/etc/sysctl.d/007-forward.conf`)
- `iptables -I FORWARD -i bond0 -o <outif> -j ACCEPT`
- `iptables -I FORWARD -i <outif> -o bond0 -m state --state RELATED,ESTABLISHED -j ACCEPT`
- `iptables -t nat -A POSTROUTING -s 10.7.0.0/24 -o <outif> -j MASQUERADE`
- Updates all peers to `allowed-ips 0.0.0.0/0`

`setup-wg.sh` re-applies gateway mode on restart if `BOND_GATEWAY=on` is set.

### Kamailio requirements

SIP clients coming through the bond tunnel are NATted. Kamailio must include:
- `force_rport()` -- forces SIP responses to the source port of the request
- `fix_nated_contact()` -- rewrites the Contact header with the actual source IP:port

Without these, SIP responses go to the wrong address and RTP negotiation fails.

## Deployment Details

### Dependencies

The installers require: `wireguard-tools`, `curl`, `jq`, `xxd`. The `xxd` package is needed for converting WireGuard public keys to hex for UAPI socket commands.

### Go Runtime Tuning

The systemd service sets:
- `GOGC=200` -- reduces GC frequency (default is 100), trading memory for lower CPU
- `GOMEMLIMIT=64MiB` -- caps total Go heap to prevent unbounded growth

### RuntimeDirectory

The systemd service uses `RuntimeDirectory=wireguard` to ensure `/var/run/wireguard/` exists at startup. Without this, the UAPI socket (`/var/run/wireguard/bond0.sock`) cannot be created and `add-bond-paths.sh` fails.

### Peer Persistence

Enrolled peers are saved to `/etc/007/peers/<tunnel_ip>` (two lines: public key, tunnel IP). `setup-wg.sh` loads all persisted peers on startup, ensuring they survive service restarts.

## rtesip Integration

The rtesip installer (`deploy/install.sh`) includes an optional 007 Bond client installation:

1. Checks if 007 Bond is already installed (`/etc/007/.env`)
2. Prompts for enrollment server URL and token (interactive)
3. If provided, downloads and runs the 007 client installer with `ENROLL_URL` and `ENROLL_TOKEN`
4. Adds systemd dependency to rtesip service: `After=007-bond.service`, `Wants=007-bond.service`
5. This ensures the bond tunnel is established before SIP registration starts
